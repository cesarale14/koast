koast brand · motion vocabulary v1.0

A layered motion system.

Three motion registers — idle, active, milestone — plus a small-size fallback. Each register has a distinct gesture and trigger condition. The motion does real work: it tells the user what Koast is doing, when something happens, and when it's at rest.

v1.0 · 2026.04.30 · supersedes motion-philosophies.html · canonical reference
register 1

Idle — static

No motion. The composed brand mark at rest. This is the default state for any deployment of the banded-O when no work is happening: app header, browser favicon, email signature, document watermark, anywhere the mark just exists.

light
dark

Why static: with cascade as a strong active signal, breath at 1-2% adds nothing — it either doesn't register (invisible to most users) or competes with cascade for "motion = active" semantics. Static is most defensible, conventional (Stripe, Linear, Vercel, Anthropic all ship static marks), and accessible by default.

register 2

Active — cascade

Bands fade through in top-down sequence. Reads as consulting layers of memory — the AI co-host is reading, computing, processing. This is the production "thinking" indicator. Triggered while AI generation is in flight, sync is running, or any user-perceptible background work is happening.

register 2b

Active fallback — pulse

At 16-31px, cascade's per-band opacity changes are too subtle to perceive. The pulse fallback affects the entire mark via CSS filter — a brightness/saturation lift that reads as "the mark is lit up while thinking." Visible at small sizes because color/brightness changes survive any pixel resolution.

register 3

Milestone — deposit

A single ghost band drops in from above; the existing stack shifts down a full band-height to make room; everything settles. Reads as "memory just deposited" — Koast just stored something significant. This is a celebration gesture, fired only on specific positive events. It's a one-shot in production, not a loop.

Trigger taxonomy — what fires deposit

The brand layer defines which events deserve the gesture; the product-side state machine that fires them lives in koast/koast-workers and is implemented as a follow-up. The list below is the locked recommendation.

tiereventcadence
Tier 1
Host milestones		First listing onboardedOnce per host
First booking receivedOnce per host
First AI-drafted message approvedOnce per host
First public reply publishedOnce per host
First successful guest review submittedOnce per host
Tier 2
System milestones		Daily pricing sync completed~once/day
Weekly reviews wrap-up sentOnce/week
Monthly summary email deliveredOnce/month
Major bug fix or feature launch (admin manual trigger)Rare
Silent
(do not fire deposit)		Every individual booking after the firstsilent
Every individual AI message after the firstcascade only
Routine API calls / polling / background syncsilent
Errors, warnings, failuresdifferent visual

The "rare, deserves attention" rule. Tier 1 fires once per host — a host who has been active for two years should never see a Tier 1 deposit again, ever. Tier 2 fires on natural cadence (daily/weekly/monthly), so a typical active host sees roughly 1-3 deposits per day. If you find yourself firing the deposit on every booking, every message, or every pricing recommendation, the bar's too low and the gesture loses its meaning. Routine events get the cascade (active state) only, which is doing a different job.

Implementation note for product code: the deposit gesture is fired by the React component when its data-state attribute transitions from idle to milestone. The component listens for animationend and transitions back to idle. Backend services emit milestone events to a dedicated event bus; the frontend subscribes via WebSocket and triggers state changes. Tier 1 events should be deduplicated via persistent storage — a host who completes "first booking" should never see that deposit again, even across sessions, devices, or account re-logins.

Marketing hero — continuous cascade

For one specific surface — the public marketing landing hero — the cascade runs continuously without rest. Same gesture, faster tempo, no quiet phase. First-time visitors evaluating Koast see motion as discovery, not fatigue.

Single context only. The hero variant lives on the public landing page hero, nowhere else. Don't propagate ambient motion to the body, pricing page, blog, or any in-app surface. The hero earns motion because users haven't committed yet — once they're in the app, they need calm.

surfaces · static only

Where motion does not live.

Active state with motion is for live web/app contexts only. The motion vocabulary does its work on the actual site, not on share previews, app launchers, or printed material. The following surfaces are locked as pure static — all bands at full opacity, evenly stacked, no motion implied.

surfaceformatrule
OG / social cards1200×630, 1080×1080Pure static. Twitter/LinkedIn/Slack compress aggressively; static survives, cascade-still might compress into confusion.
iOS app icon1024×1024 rounded squareStatic. App launchers don't render animation; motion would never be seen.
Android adaptive icon432×432 foreground + bgStatic.
Windows tile310×310Static.
Apple touch icon180×180Static.
Favicon (192, 512)PNGStatic. Browser tab favicons render at 16px regardless; large variants exist for PWA install only.
Favicon (16, 24, 32, 48)PNG / .icoStatic. Below 16px no animation possible; favicons should not animate at any size as a category rule.
Print / PDF brand assetsanyStatic. Obviously.
Email signaturesPNG / SVG staticStatic. Most email clients block animated content.

Why this rule matters. First-time viewers seeing the brand on a share preview need the cleanest read of what Koast is. A cascade-still — a single frame mid-fade — would look like a degraded version of the static composition. The bands would look unevenly weighted. The motion vocabulary earns its place in live product surfaces because users see it across time; in a single frame, motion is invisible at best, broken-looking at worst.

State transitions

How states move into and out of one another. The mark has three runtime states: idle, active, milestone. Transitions are handled via a data-state attribute on the mark element; CSS handles the visual change.

fromtotriggervisual
idleactiveAsync work starts (AI generating, sync running, etc.)Cascade begins on next animation frame
activeidleAsync work completesCascade fades; mark settles to static via opacity transition
idlemilestoneTier 1 or Tier 2 event firesOne-shot deposit plays; animationend returns to idle
activemilestoneRare — milestone event fires while still activeActive completes current cycle, milestone plays, returns to idle (cancels active even if work continues — motion budget rule)
milestoneidleDeposit animation completesSingle transition; clean settle

Implementation pattern (React)

function KoastMark({ size = 96 }) {
  const state = useKoastState(); // 'idle' | 'active' | 'milestone'

  return (
    <span
      className="koast-mark"
      data-state={state}
      data-size={size < 32 ? 'small' : 'normal'}
    >
      <svg viewBox="0 0 100 100" width={size} height={size}>
        {/* 5 band rects with .b1...b5 classes */}
        {/* + .ghost rect at y=-23 for milestone */}
      </svg>
    </span>
  );
}

Reduced-motion compliance

@media (prefers-reduced-motion: reduce) {
  .koast-mark .b1,
  .koast-mark .b2,
  .koast-mark .b3,
  .koast-mark .b4,
  .koast-mark .b5,
  .koast-mark .ghost,
  .koast-mark .stack,
  .koast-mark .bands {
    animation: none !important;
  }
}

When users opt out of motion, all three registers collapse to static. Active and milestone states should fall back to a non-motion indicator if a status signal is required (e.g. a small loading spinner adjacent to the mark, or a text status label).
